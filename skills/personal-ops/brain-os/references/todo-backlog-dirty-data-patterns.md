# todo-backlog 脏数据模式

## 问题描述

史官等生产者任务在不同日期重复记录同一问题，导致 todo-backlog.md 中存在大量重复/碎片化条目。

## 常见模式

### 模式 1：同一问题多次记录

**示例**：
```
| H001 | `728448c1` 史官 | LLM Wiki 相关待办未同步到 todo | open | - | 2026-05-18 | 2026-05-18 | - | - |
| H002 | `728448c1` 史官 | LLM Wiki 待办同步问题 | open | - | 2026-05-19 | 2026-05-19 | - | - |
| H003 | `728448c1` 史官 | LLM Wiki 待办未写入 todo | open | - | 2026-05-19 | 2026-05-19 | - | - |
```

**影响**：阻塞项统计虚高（3 条实际仅 1 条有效）

### 模式 2：Telegram 推送错误重复

**示例**：
```
| M001 | `3cda2159` 观察者 | Telegram 推送失败 | open | - | 2026-05-18 | 2026-05-18 | - | - |
| M002 | `3cda2159` 观察者 | Telegram 推送错误 | open | - | 2026-05-19 | 2026-05-19 | - | - |
```

**影响**：同一错误被重复记录

### 模式 3：钉钉鉴权失败重复

**示例**：
```
| M003 | `3cda2159` 观察者 | 钉钉 Stream API 鉴权失败 | open | - | 2026-05-18 | 2026-05-18 | - | - |
| M004 | `3cda2159` 观察者 | 钉钉鉴权 401 错误 | open | - | 2026-05-19 | 2026-05-19 | - | - |
```

**影响**：同一鉴权问题被重复记录

### 模式 4：WebUI 构建超时碎片化

**示例**：
```
| L001 | `ce8a1963` 巡检 | WebUI 构建超时 - 任务 ID | open | - | 2026-05-18 | 2026-05-18 | - | - |
| L002 | `ce8a1963` 巡检 | WebUI 构建超时 - 路径问题 | open | - | 2026-05-18 | 2026-05-18 | - | - |
| L003 | `ce8a1963` 巡检 | WebUI 构建超时 - Git 问题 | open | - | 2026-05-18 | 2026-05-18 | - | - |
| L004 | `ce8a1963` 巡检 | WebUI 构建超时 - 命令问题 | open | - | 2026-05-18 | 2026-05-18 | - | - |
| L005 | `ce8a1963` 巡检 | WebUI 构建超时 - 超时问题 | open | - | 2026-05-18 | 2026-05-18 | - | - |
```

**影响**：一个问题的多个方面被拆成 5 条独立记录

## 检测方法

```bash
# 按主题关键词聚类检测重复
grep -E "TODO-[0-9]+-" todo-backlog.md | sed 's/|.*//' | sort | uniq -c | sort -rn

# 检测同一来源任务的重复
grep "| \`728448c1\` 史官" todo-backlog.md | wc -l

# 检测 open 状态数量
grep -c "| open |" todo-backlog.md
```

## 修复策略

### 策略 1：合并重复条目

1. 找出所有重复条目（相同主题 + 相同来源）
2. 保留最新一条
3. 删除旧条目

### 策略 2：合并碎片化条目

1. 找出同一问题的多个碎片化记录
2. 合并为单一条目，描述涵盖所有方面
3. 删除碎片化记录

### 策略 3：标记已解决但未更新状态的条目

1. 检查 Kanban 卡片状态
2. 如果 Kanban 已 resolved，更新 todo 状态为 resolved
3. 写入完成时间和处理方式

### 策略 4：史官 prompt 改造

在史官记录 prompt 中增加检查步骤：

```
在记录待办前：
1. 检查 todo-backlog.md 中是否已存在相同主题
2. 如果已存在，更新现有条目而非创建新条目
3. 如果不存在，创建新条目
```

## 预防措施

### 生产者 prompt 规范

所有生产者任务应在 prompt 中明确：

```
在写入 todo 前：
1. 读取 todo-backlog.md
2. 检查是否已存在相同问题（通过问题摘要相似度 + 来源任务）
3. 如果已存在，更新现有条目（更新最后更新时间、添加备注）
4. 如果不存在，创建新条目
```

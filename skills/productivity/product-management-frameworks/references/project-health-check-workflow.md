# 项目全面审计工作流 (Project Health Check)

当用户要求"用项目管理技能制定项目计划"时，执行以下多框架链式工作流：

## 触发条件
- 用户说"制定项目计划"、"推进项目"、"全面管理项目"
- 用户要求使用 PM 技能对现有项目进行审计
- 新项目启动需要完整性评估

## 执行步骤

### Step 1: 数据采集 — 摸清现状
从 Paperclip 拉取完整数据：
```
- Project 信息（名称、ID、状态、描述）
- Goals 层级（顶级 + 子级，区分 active/completed）
- Agents 列表（ID、名称、角色、适配器、状态）
- Issues 全量（按 status 统计分布 + goal 关联）
```

### Step 2: 健康诊断
计算关键指标，给出评级：
| 维度 | 计算方法 | 评级标准 |
|------|----------|----------|
| 完成率 | done / total | >30% ✅ / 15-30% 🟡 / <15% 🔴 |
| 阻塞率 | blocked / total | <10% ✅ / 10-25% 🟡 / >25% 🔴 |
| 推进率 | in_progress / total | >20% ✅ / 10-20% 🟡 / <10% 🔴 |

### Step 3: SWOT 分析
加载 `swot-analysis` 技能，定位：
- **Strengths**: 已有成果、稳定流程、技术积累
- **Weaknesses**: 阻塞率、技能缺失、零客户验证
- **Opportunities**: 市场刚需、先发优势、可用日报做MVP
- **Threats**: 基础设施风险、大厂入场、客户付费意愿

战略建议：
- **Build**: 利用优势捕捉机会
- **Defend**: 修补弱点抵御威胁
- **Fix**: 优先解决阻塞性问题

### Step 4: Pre-mortem 风险预判
加载 `pre-mortem` 技能，做三类分类：
- 🐅 **Tigers**（真实风险）：影响核心功能的问题，需立即行动
- 🐘 **Elephants**（未提及但重要）：团队回避讨论的隐忧，需要用户决策
- 🧻 **Paper Tigers**（过度担忧）：可以忽略，节省精力

### Step 5: 需求完整性评估（逐个问，不要一次性抛）\n\n**面谈方式**：一次只问一个缺口，给出选项让用户选，用户回答后记录再问下一个。不要一次性列出所有问题让用户消化。\n\n检查以下 5 个维度是否覆盖：\n1. **客户是谁**（ICP）→ 目标客户细分 + 未来扩展路径\n2. **产品形态**（SaaS/推送/咨询）→ 核心交付方式 + 补充渠道\n3. **价值主张**（省钱/赚钱/省时间）→ 哪一项最打动客户\n4. **竞争对手**（替代方案/差异化）→ 竞品是工具还是服务 | 我们的差异点\n5. **成功标准**（北极星指标）→ 阶段化目标（内测→试点→商业化）\n\n**典型用户回答参考**：\n- 客户是谁：当前做A客户 → 稳定后扩展B → 未来到C（三层扩展路径）\n- 产品形态：SaaS网页为主 + 多渠道推送为补充\n- 价值主张：帮客户赚钱（找商机）> 帮客户避险\n- 差异化：竞品是查询工具，我们是分析决策引擎

### Step 6: 制定里程碑
按 Phase 0-4 组织输出：
```
Phase 0: 紧急修复（本周）
Phase 1: 需求调研（需要用户参与）
Phase 2: 产品化（基于调研结果）
Phase 3: 市场验证
Phase 4: 规模化
```

### Step 7: 问决策
给出两条路径让用户选择：
- **A) 立即推进** — 现有方向继续，承担"客户不买单"风险
- **B) 先做需求调研** — 用户作为PM，30分钟引导补齐缺口

### Step 8: 沉淀到系统\n- Wiki: `05-项目/{项目名}/计划文档/项目全面审计与计划-YYYY-MM-DD.md`\n- Paperclip: 更新 Goals/Issues 反映新计划\n  - API认证：需要 cookie + `Origin: http://localhost:3100` 头才能写操作\n  - Issue 创建后获取 identifier（如 CMP-58）用于跨系统关联\n\n## 产品数据录入模式（用户无法访问ERP时）\n\n当用户说\"看不到具体产品信息\"时，给三个递进方案：\n\n**方案A：HS编码展开法**（推荐）\n- 用公开 HS 编码体系将用户给的品类名展开到 6-8 位子目\n- 不做精确匹配，做到品类级即可分析\n- 优点是立即能跑，后续可校准\n\n**方案B：口头补充法**\n- 用户凭记忆给出 1-2 个常做品种\n- 你补充 HS 编码 + CAS号 + 英文名\n- 优点是最精确，但依赖用户记忆\n\n**方案C：宏观分析法**\n- 直接用 HS 2位章级跑全球贸易数据\n- 回答\"哪个区域大量进口这类产品\"级别的宏观问题\n- 精度最低但最快出方向性结论\n\n**推荐策略**：先 A+C 组合跑第一轮，出有价值结果后让用户校准。别等用户凑齐数据才启动，先做有价值的事。

## 关键原则
- **先审计再行动**：不评估现状就做计划是拍脑袋
- **风险优先于功能**：阻塞问题不修，全部功能都是纸面功夫
- **用户决策不可跳过**：需求缺口必须由用户（产品经理）决定
- **框架链比单个框架更有效**：SWOT → Pre-mortem → Gap Analysis 的链式组合

# Knowledge Base Audit: Methodology & Pitfalls

## Methodology from 2026-05-18 Audit

The weekly audit on 2026-05-18 scanned 495 markdown files and produced a structured report at `04-知识库/99-系统/01-索引/weekly-audit-2026-05-18.md`.

### Scan Order

1. **File discovery** — `os.walk` skipping `.git`
2. **Basename/title maps** — for resolving Obsidian `[[wikilink]]` to actual file path
3. **YAML frontmatter** — check `---` delimiters + key-value content
4. **Wikilink resolution** — check each `[[target]]` against title map, then fallback to basename map
5. **Inbound link graph** — build reverse map from wikilink sources
6. **Duplicate basename detection** — group by filename stem
7. **File size audit** — detect <50b and >500KB outliers

### Key Pitfalls Encountered

#### 1. index.md Directory Migration Artifact
The root `index.md` references `03-知识库/...` paths, but the actual directory was renamed to `04-知识库/` during the 2026-05-14 restructuring. This caused **22 broken wikilinks** in the index.

**Fix**: Update `index.md` to point to current paths. The index should be regenerated after any directory renumbering.

#### 2. SCHEMA.md False Positives
`05-系统配置/SCHEMA.md` uses `[[wikilinks]]` as a literal example in documentation. This is not a real broken link. Any audit must filter SCHEMA.md explicitly.

#### 3. Cron Prompt Template Paths
Prompt files (`05-系统配置/提示词/cron/*`) contain placeholder wikilinks like `[[03-KNOWLEDGE/01-READING/04-DIGESTS/nightly-digest-YYYY-MM-DD]]`. These are runtime-substituted templates, not broken links.

#### 4. Orphan Note Data Interpretation
446 files had no inbound links, but ~90% are auto-generated content:
- Short drama scripts (70+)
- Channel history logs (120+)
- Nightly digest/reports (50+)
- Agent config files

The ~30 genuinely orphaned knowledge documents are in `04-知识库/01-阅读消化/01-领域知识/` and `06-工作上下文/数字化运维/`. These should be linked from an appropriate index.

#### 5. Duplicate Basename Interpretation
57 groups of duplicate basenames fell into three categories:
- **Type A — True duplicates**: Same file content stored in multiple locations (e.g., risk alert daily reports in 4 places). Need consolidation.
- **Type B — Intentional mirrors**: Deliverable in `02-工作产出/` + knowledge copy in `04-知识库/`. Acceptable by design.
- **Type C — Same name, different content**: `episode_001` exists in two different short drama projects. Not a real duplicate.

## Baseline Expectations for This Knowledge Base

| Metric | Expected | Notes |
|--------|----------|-------|
| Files with YAML frontmatter | Low (~8%) | Only documents manually authored get it |
| Broken wikilinks | <5 (excluding templates/docs) | Should be 0 for real content |
| Orphan notes | High (~90%) | Auto-generated content is expected to be orphan |
| Duplicate basenames | ~3-5 groups need cleanup | Focus on Type A duplicates |
| Small/empty files | 0 | Should always be 0 |
| Large files (>500KB) | 0 | Knowledge content is text-based |

## Audit Output Distribution

- **Full report**: `04-知识库/99-系统/01-索引/weekly-audit-YYYY-MM-DD.md`
- **Lint report** (auto-generated by cron): `99-系统/lint-reports/lint-report-YYYY-MM-DD.md`
- **Audit script**: `scripts/kb-audit.py` (under knowledge-management-system skill)

The lint report is focused on format issues (trailing whitespace, long lines). The audit report is focused on structural health (frontmatter, links, orphans).
